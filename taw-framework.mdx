## What counts as a tool in TAW Core

TAW Core tools are the developer-facing building blocks the theme uses: command-line utilities and runtime PHP APIs.

These tools fall into two groups:

- **CLI commands** — Symfony Console commands exposed through the `bin/taw` script in your theme. Use them to generate and manage blocks and other theme artifacts.
- **Runtime APIs and helpers** — PHP classes, functions, and configuration points that run inside WordPress, such as theme boot configuration, performance tuning, block registration, admin UI engines, config-driven forms, mail templates, and debugging helpers.

<Callout kind="info">

TAW Theme builds on top of these TAW Core primitives. When you configure the theme or call theme helpers, you are usually driving these lower-level tools under the hood.

</Callout>

## How TAW Core boots into the theme

The theme template boots TAW Core via the `TAW\Core\Theme::boot()` entrypoint. You can then adjust performance behavior with `Theme::performance([...])`.

This boot process wires TAW Core tools into WordPress so that CLI commands, block registration, metaboxes, and other helpers are available within your theme.

## New runtime modules: Forms and Mail

Forms and Mail in TAW Core give you a config-driven way to handle front-end forms, store submissions, post them to webhooks, and send templated emails using your theme's mail layouts.

### Forms: config-driven builder and submissions

Use `TAW\Core\Form\Form` to define and render forms from configuration, with built-in validation, honeypot/CSRF patterns, and submission saving.

A minimal pattern inside a template or controller looks like this:

```php
use TAW\Core\Form\Form;

$form = new Form([
    'id' => 'contact',
    'fields' => [
        // Define your fields here using the Form configuration schema.
    ],
    'email' => [
        // Configure email behavior for submissions here.
    ],
]);

echo $form->render();
```

`Form` coordinates rendering, validation, and saving the submission to the `taw_submission` custom post type through the submissions handler.

`TAW\Core\Form\SubmissionsHandler` registers the `taw_submission` post type and optionally posts each submission to a configured webhook.

<Callout kind="info">

Configure the webhook endpoint and HMAC secret under **Settings > Form Webhook** in the WordPress admin. TAW Core signs outbound submission payloads so your receiver can verify authenticity.

</Callout>

### Mail: templates and fluent sending

Use `TAW\Core\Mail\MailTemplate` to work with HTML or MJML templates from your theme and `TAW\Core\Mail\Mailer` to send emails with variable replacement.

Mail templates live under your theme, for example `mails/contact.html` or `mails/contact.mjml`. `MailTemplate::getAvailable()` exposes the templates that exist on disk, and each template supports `{{var}}` placeholders for variables you pass at send time.

A minimal example of sending a templated email:

```php
use TAW\Core\Mail\Mailer;

$mailer = new Mailer();

$sent = $mailer
    ->to('support@acme-agency.test')
    ->subject('New contact form submission')
    ->template('contact') // resolves mails/contact.html or .mjml
    ->setVariables([
        'name' => 'Jane Chen',
        'email' => 'jane@acme.com',
        'message' => 'I would like to discuss a new project.',
    ])
    ->send();

if (! $sent) {
    // Handle failed mail transport (log, retry, etc.).
}
```

<Callout kind="info">

`Mailer` uses the underlying `wp_mail()` transport. Make sure your WordPress site is configured with a working mail provider (SMTP or transactional service) so test and production emails are delivered reliably.

</Callout>

`TAW\Core\Mail\MailTemplate` compiles templates from your theme, performs `{{var}}` replacement, and is responsible for producing the final HTML payload that `Mailer` sends.

### Admin entrypoints and testing

TAW Core adds admin screens to help you operate and test these modules without writing ad-hoc scripts.

- **Tools > Test Emails** — registered by `MailTester::register()`, lets you send test emails using your templates and variables to verify layout and delivery in your environment.
- **Settings > Form Webhook** — configures the webhook URL and HMAC secret that `SubmissionsHandler` uses when posting saved submissions to an external endpoint.

Use these screens to confirm templates render as expected and form submissions reach your downstream systems before wiring them into live flows.

## When to reach for each tool

Choose tools based on the kind of change you are making:

- **New or updated content blocks** — Use the **TAW CLI** (`make:block`) and the **Blocks API** for registration and rendering.
- **Admin UX or content modeling** — Use the **Metabox engine** and **Options pages** to add structured configuration in the admin.
- **Performance or infrastructure changes** — Adjust **Theme boot and performance** configuration and wire assets through the **Vite/asset pipeline**.
- **Troubleshooting** — Turn on and use **Debugging utilities** to inspect what TAW Core is doing at runtime. The enhanced `dump()` / `dd()` helpers format output cleanly and are intended for local or development use only, not production.
